# Тема 1: Массивы, строки и хэш-таблицы

> **Главная идея темы:** хэш-таблица позволяет быстро отвечать на вопросы вида «видел ли я уже этот элемент?», «сколько раз он встречался?», «где он был раньше?». За счёт этого многие задачи улучшаются с O(n²) до O(n).

---

## Что здесь главное

В задачах этой темы почти всегда есть один из трёх вопросов:

1. Нужно ли **посчитать** элементы?
2. Нужно ли **быстро проверить наличие** элемента?
3. Нужно ли **сгруппировать эквивалентные** объекты?

Если да — почти наверняка поможет `Map`, `Set` или объект-словарь.

---

## Как работает хэш-таблица

Хэш-таблица (`Map` / `Object` / `Dict`) хранит пары `ключ -> значение` и позволяет в среднем за O(1) выполнять:

| Операция | Средняя сложность | Худший случай |
| -------- | ----------------- | ------------- |
| Вставка  | O(1)              | O(n)          |
| Поиск    | O(1)              | O(n)          |
| Удаление | O(1)              | O(n)          |

На интервью почти всегда рассматривают средний случай.

**Set** — это та же идея, но без значений: он нужен, когда важно только «есть / нет».

---

## 3 базовых паттерна

### 1. Frequency Counter

**Когда применять:**  
Когда нужно понять, сколько раз встречается каждый элемент.

**Типичные формулировки:**

- проверить, являются ли строки анаграммами;
- найти дубликаты;
- сравнить состав двух наборов данных;
- найти самые частые элементы.

**Идея:**  
Строим карту вида:

```text
элемент -> количество
```

**Общий шаблон:**

```javascript
const freq = new Map();

for (const elem of data) {
  freq.set(elem, (freq.get(elem) ?? 0) + 1);
}
```

**Пример — Valid Anagram (LC 242):**

```javascript
function isAnagram(s, t) {
  if (s.length !== t.length) return false;

  const freq = new Map();

  for (const ch of s) {
    freq.set(ch, (freq.get(ch) ?? 0) + 1);
  }

  for (const ch of t) {
    if (!freq.has(ch) || freq.get(ch) === 0) return false;
    freq.set(ch, freq.get(ch) - 1);
  }

  return true;
}
// Время: O(n)
// Память: O(1) при фиксированном алфавите, иначе O(n)
```

**Ключевая мысль:**  
Вместо того чтобы для каждого символа заново искать совпадение, мы один раз считаем частоты и потом сравниваем их.

### 2. Presence / Lookup

**Когда применять:**  
Когда нужно быстро проверять, встречался ли элемент раньше, или находить пару к текущему элементу.

**Типичные формулировки:**

- «найди два числа, дающих сумму X»;
- «проверь, есть ли дубликаты»;
- «найди самую длинную последовательность»;
- «проверь, существует ли нужное значение».

**Идея:**  
Храним в структуре то, что может понадобиться позже:

```text
элемент -> индекс
элемент -> true
нужное значение -> данные
```

#### Частный случай: Complement Search

Это подтип `Presence / Lookup`.

Если для текущего элемента нужен комплемент, например:

```text
complement = target - num
```

то вместо полного перебора мы просто проверяем, был ли этот `complement` раньше.

**Пример — Two Sum (LC 1):**

```javascript
function twoSum(nums, target) {
  const seen = new Map();

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];

    if (seen.has(complement)) {
      return [seen.get(complement), i];
    }

    seen.set(nums[i], i);
  }
}
// Время: O(n)
// Память: O(n)
```

#### Частный случай: Set + проверка начала последовательности

Если задача связана с последовательностями, диапазонами или уникальностью, часто достаточно `Set`.

Идея в том, чтобы начинать расширение только из **начала** последовательности.  
Число `num` — начало цепочки, если `num - 1` отсутствует в множестве.

**Пример — Longest Consecutive Sequence (LC 128):**

```javascript
function longestConsecutive(nums) {
  const set = new Set(nums);
  let maxLen = 0;

  for (const num of set) {
    if (!set.has(num - 1)) {
      let curr = num;
      let len = 1;

      while (set.has(curr + 1)) {
        curr++;
        len++;
      }

      maxLen = Math.max(maxLen, len);
    }
  }

  return maxLen;
}
// Время: O(n)
// Память: O(n)
```

**Ключевая мысль:**  
Если структура отвечает на вопрос «есть ли это значение?» за O(1), очень много задач на поиск и проверку становятся линейными.

### 3. Grouping by Canonical Key

**Когда применять:**  
Когда нужно сгруппировать объекты, которые эквивалентны по смыслу, но выглядят по-разному.

**Типичные формулировки:**

- сгруппировать анаграммы;
- сгруппировать одинаковые наборы символов;
- объединить объекты по нормализованной форме.

**Идея:**  
Нужно придумать **канонический ключ** — одинаковое представление для всех объектов одной группы.

```text
нормализованный_ключ -> группа
```

**Примеры канонического ключа:**

- отсортированная строка;
- вектор частот;
- строка после нормализации регистра и удаления лишних символов.

**Пример — Group Anagrams (LC 49):**

```javascript
function groupAnagrams(strs) {
  const groups = new Map();

  for (const str of strs) {
    const key = str.split("").sort().join("");

    if (!groups.has(key)) {
      groups.set(key, []);
    }

    groups.get(key).push(str);
  }

  return [...groups.values()];
}
// Время: O(n * k log k)
// Память: O(n * k)
```

**Ключевая мысль:**  
Если две сущности должны попасть в одну группу, нужно научиться превращать их в один и тот же ключ.

## Производные техники

Это не отдельные базовые паттерны, а комбинации базовых идей.

### Top K Frequent = Frequency Counter + извлечение ответа

Сначала считаем частоты, затем выбираем способ достать топ-K:

- bucket sort, если нужен O(n);
- heap, если нужен более общий подход;
- сортировка, если ограничений мало.

**Пример — Top K Frequent Elements (LC 347):**

```javascript
function topKFrequent(nums, k) {
  const freq = new Map();

  for (const num of nums) {
    freq.set(num, (freq.get(num) ?? 0) + 1);
  }

  const buckets = Array.from({ length: nums.length + 1 }, () => []);

  for (const [num, count] of freq) {
    buckets[count].push(num);
  }

  const result = [];

  for (let i = buckets.length - 1; i >= 0 && result.length < k; i--) {
    result.push(...buckets[i]);
  }

  return result.slice(0, k);
}
// Время: O(n)
// Память: O(n)
```

**Важно:**  
`Bucket Sort` здесь — не самостоятельная идея темы, а способ обработать уже посчитанные частоты.

## Как быстро распознать паттерн

При чтении условия задай себе вопросы:

```text
1. Нужно ли знать, сколько раз встречается каждый элемент?
   -> Frequency Counter

2. Нужно ли быстро проверять, есть ли элемент или видели ли мы его раньше?
   -> Presence / Lookup

3. Нужно ли найти пару, дополнение или нужное значение к текущему элементу?
   -> Presence / Lookup -> Complement Search

4. Нужно ли сгруппировать эквивалентные объекты?
   -> Grouping by Canonical Key

5. Нужно ли найти top-K по частоте?
   -> Frequency Counter + способ извлечения (bucket / heap / sort)
```

## Универсальные шаблоны

### Шаблон A — один проход

Подходит для задач вроде `Two Sum`.

```text
for element in data:
    need = f(element)

    if need already seen:
        return answer

    save current element
```

### Шаблон B — сначала строим, потом используем

Подходит для задач вроде `Valid Anagram`.

```text
1. Строим карту/множество
2. Вторым проходом получаем ответ
```

### Шаблон C — нормализуй и группируй

Подходит для задач вроде `Group Anagrams`.

```text
for object in data:
    key = canonical_form(object)
    groups[key].push(object)
```

## Типичные ошибки

| Ошибка                                         | Как правильно                                 |
| ---------------------------------------------- | --------------------------------------------- |
| Забыть раннюю проверку длины в анаграмме       | Сначала проверить `s.length !== t.length`     |
| Использовать обычный объект для сложных ключей | Для надёжности использовать `Map`             |
| Проверять значение через truthy/falsy          | Лучше использовать `map.has(key)`             |
| Думать, что `Set` и `Map` занимают O(1) памяти | По памяти это O(n)                            |
| Считать `Bucket Sort` отдельным hash-паттерном | Это производная техника после подсчёта частот |

## Сложность популярных задач

| Задача                       | Паттерн                               | Время           | Память        |
| ---------------------------- | ------------------------------------- | --------------- | ------------- |
| Two Sum (LC 1)               | Presence / Lookup + Complement Search | O(n)            | O(n)          |
| Valid Anagram (LC 242)       | Frequency Counter                     | O(n)            | O(1)\* / O(n) |
| Group Anagrams (LC 49)       | Canonical Key                         | O(n \* k log k) | O(n \* k)     |
| Top K Frequent (LC 347)      | Frequency Counter + Bucket Sort       | O(n)            | O(n)          |
| Longest Consecutive (LC 128) | Presence / Lookup + Set               | O(n)            | O(n)          |

\* O(1), если алфавит фиксирован.

---

## Что говорить на интервью

Удобная структура ответа:

1. **Brute force:**  
   «Наивное решение — перебор, обычно O(n²).»

2. **Идея оптимизации:**  
   «Можно сохранить уже встреченные элементы / их частоты в хэш-таблице и проверять нужное значение за O(1).»

3. **Почему это работает:**  
   «Мы не ищем повторно по всему массиву, а делаем быстрый lookup.»

4. **Сложность:**  
   «Получаем O(n) по времени и O(n) по памяти.»

5. **Edge cases:**  
   пустой ввод, дубликаты, отрицательные числа, разные длины строк, `k > n`.

---

## Порядок практики

Лучше идти так:

1. `Valid Anagram` — понять `Frequency Counter`
2. `Two Sum` — понять `Presence / Lookup`
3. `Group Anagrams` — понять `Canonical Key`
4. `Longest Consecutive Sequence` — понять силу `Set`
5. `Top K Frequent Elements` — научиться комбинировать `Frequency Counter` с извлечением ответа

Такой порядок лучше, потому что каждая следующая задача строится на предыдущих идеях.
